Project Overview

Purpose

This guide establishes shared engineering standards for delivering maintainable, secure, and scalable web products with consistent quality across teams.

Typical Tech Stack

  • Next.js (App Router), React, TypeScript
  • State: Redux Toolkit / React Query (as appropriate)
  • Styling: Tailwind CSS or CSS Modules
  • Lint/Format: Biome (or ESLint + formatter equivalent)
  • CI/CD: GitHub Actions + cloud deployment platform

Scope

Apply this document to all frontend, API, and infrastructure-facing changes unless a repository-specific standard supersedes it.

Architecture

Recommended high-level architecture

  • Presentation Layer: routes, pages, and UI components
  • Feature Layer: domain-specific behavior and orchestration
  • Service Layer: API clients, adapters, and external integrations
  • Shared Layer: reusable UI, utility functions, and type contracts

Principles

  • Keep boundaries explicit and dependencies directional.
  • Prefer Server Components by default; use Client Components intentionally.
  • Move business logic out of route files into testable modules.
Rule of thumb: Route files should compose and coordinate; services and features should implement behavior.

Folder Structure

.
├─ app/                    # Routes, layouts, route handlers
├─ components/             # Shared presentational components
├─ features/               # Domain modules (feature-first)
├─ services/               # External API/data providers
├─ lib/                    # Utilities and cross-cutting helpers
├─ store/                  # Global client state (if needed)
├─ types/                  # Shared TypeScript contracts
├─ styles/                 # Global/theme styling
└─ tests/                  # Integration/e2e suites (optional location)

Naming conventions

  • Components: PascalCase.tsx (e.g., FlightResults.tsx)
  • Helpers/services: descriptive camelCase.ts or kebab-case.ts
  • Tests: *.test.ts / *.test.tsx
  • Barrel exports: index.ts at package/feature boundaries

Coding Standards

Core standards

  • Optimize for readability and future maintainers.
  • Use strict typing and explicit contracts.
  • Favor small, focused functions and composable modules.
  • Avoid hidden side effects and implicit behavior.

File organization

  1. Imports (external, aliases, relative)
  2. Types/interfaces
  3. Constants/helpers
  4. Main implementation
  5. Exports

Formatting

  • Enforce with automated tooling in CI and pre-commit.
  • Do not manually reformat unrelated files in functional PRs.
# Example quality gate
pnpm lint
pnpm check
pnpm build

Git Workflow

Branch strategy

  • main: releasable trunk
  • feature/<topic>: feature development
  • fix/<topic>: bug fixes
  • hotfix/<topic>: urgent production corrections

Commit conventions

Use Conventional Commits:

feat(auth): add token refresh strategy
fix(api): validate required query params
docs(guide): clarify branching model

Pull request standards

  • State problem, approach, risks, and tradeoffs.
  • Attach screenshots for UI changes.
  • Include validation evidence (lint, tests, build).
  • Keep PRs focused and reviewable.

API Guidelines

Design rules

  • Use predictable REST semantics and clear resource naming.
  • Keep handlers thin; delegate to services/use-cases.
  • Return stable JSON envelopes.
{
  "data": { "id": "f_123" },
  "error": null,
  "meta": { "requestId": "req_abc" }
}

Error handling

  • Map domain failures to correct status codes (400/401/403/404/422/500).
  • Do not leak stack traces to clients.

Validation and logging

  • Validate body/query/headers at API boundaries.
  • Log structured context (without secrets/PII).

Testing

Unit testing

  • Test pure logic and edge cases first.
  • Mock network/time/randomness for deterministic runs.

Integration testing

  • Verify critical user flows and API contracts.
  • Cover feature orchestration across module boundaries.

Naming example

describe('searchFlights')
  it('returns filtered flights when origin and destination are provided')

Security

Authentication & Authorization

  • Enforce authorization server-side for sensitive operations.
  • Use least privilege for roles and service credentials.

Data protection

  • Treat all client input as untrusted.
  • Validate and sanitize inputs and outputs.

Secrets management

  • Never commit secrets to source control.
  • Use environment variables and secret stores in CI/CD platforms.
  • Maintain an .env.example without real credentials.

Deployment

Environment model

Environment Purpose Trigger
Development Local developer iteration Manual (pnpm dev)
Preview PR validation and stakeholder review Pull request events
Production Customer-facing release Merge/push to main

CI/CD baseline

  1. Install dependencies using lockfile
  2. Run static checks and tests
  3. Build deployable artifacts
  4. Deploy and publish release metadata

Do’s and Don’ts

Do

  • ✅ Keep modules small, cohesive, and testable.
  • ✅ Document tradeoffs in PRs and architectural changes.
  • ✅ Prefer typed contracts over loosely shaped objects.
  • ✅ Automate checks in CI and pre-commit hooks.

Don’t

  • ❌ Mix routing, business rules, and integration logic in one file.
  • ❌ Add global state for purely local UI concerns.
  • ❌ Depend on manual formatting/linting discipline.
  • ❌ Expose internal error details or secrets in logs/responses.
This guide is a living document. Update it in the same pull request whenever you introduce a new architectural or coding pattern.
